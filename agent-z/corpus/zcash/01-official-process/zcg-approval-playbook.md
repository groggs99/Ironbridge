# ZCG Approval Playbook for Agent Z

Collected date: 2026-05-26
Status: derived approval insights from official ZCG process, homepage, and recent GitHub issue patterns.

## Core framing

ZCG funds independent teams that improve the Zcash ecosystem through public-good work. Agent Z should optimise proposals for clear Zcash-specific benefit, credible delivery, realistic milestones, and community-facing transparency.

## Approval funnel

```text
Idea → GitHub application → required forum post → at least 1 week community review → committee review → revisions → decision → KYC/agreement if needed → monthly updates before milestone payouts
```

## Pre-review blockers Agent Z must catch

| Blocker | Agent Z action |
|---|---|
| No forum post | Require applicant to draft and post forum version; add link to GitHub issue. |
| Budget math does not reconcile | Ensure startup funding + milestone payouts = total requested amount. |
| Grant above $50k | Flag KYC requirement and stronger due-diligence expectations. |
| Vague milestones | Convert activity into deliverables, validation criteria, and payout amounts. |
| Weak Zcash-specific benefit | Push applicant to explain why Zcash, not just privacy/Web3 generally. |
| No user validation | Identify intended users or representatives who can confirm deliverables. |
| No maintenance/update plan | Add post-grant maintenance, reporting, or sustainability section. |

## Strong proposal formula

```text
Specific Zcash problem
+ clear ZCG wishlist alignment
+ credible team
+ concrete deliverables
+ reconciled milestone budget
+ user/community validation
+ forum-ready explanation
+ realistic KPIs
+ maintenance/update plan
= approval-ready proposal
```

## Required proposal structure

```markdown
# Grant Application: [Project Name]

## One-line summary
What this does for Zcash in one sentence.

## Zcash alignment
Which ZCG wishlist category this fits and why.

## Problem
The specific Zcash ecosystem problem.

## Solution
What will be built, delivered, operated, or taught.

## Beneficiaries
Who benefits and how.

## Prior work / credibility
Evidence that the team can deliver.

## Deliverables
Concrete shipped outputs.

## Milestones
Milestone-by-milestone plan with dates, deliverables, validation, and payout amount.

## Budget
Exact total, broken down by milestone. Startup funding plus milestones must equal the requested grant amount.

## Risks
Technical, operational, adoption, governance, or community risks.

## User validation
Who will confirm the deliverables meet the required standard.

## Forum plan
Link to required forum post and plan for community updates.

## Maintenance / post-grant plan
What happens after the grant ends.
```

## Grant fit score

```text
1. Zcash alignment: /20
2. Public-good value: /15
3. Deliverable clarity: /15
4. Milestone quality: /15
5. Budget credibility: /10
6. Team credibility: /10
7. Community/forum readiness: /10
8. Risk handling: /5

Total: /100
```

Interpretation:

| Score | Meaning |
|---:|---|
| 85-100 | Strong candidate; draft application. |
| 70-84 | Viable but needs tightening. |
| 55-69 | Needs major revision before submission. |
| Below 55 | Do not submit yet. |

## Category-specific strategy

### Technical / infrastructure grants

Must include architecture, dependencies, repos, deliverables, test plan, security risks, maintenance plan, reviewers or intended users.

### Education / media grants

Must include target audience, content plan, distribution channels, KPIs, past content examples, language/region focus, proof of audience.

### Regional community grants

Must include region, local problem, why Zcash matters there, events/content/workshops, attendance targets, wallet/action metrics, local partnerships, monthly reports.

### Sponsorship / event grants

Must include event relevance, audience quality, specific Zcash activation, deliverables, post-event report, lead capture/onboarding, content outputs.

## Behaviour rule for Agent Z

Agent Z should behave like a constructive pre-submission reviewer, not a generic grant writer.

Examples:

- "This proposal is not ready because the Zcash-specific benefit is unclear."
- "The budget does not reconcile."
- "You need a forum post before review."
- "Your milestones describe work, not deliverables."
- "This asks for too much without enough proof."
- "This would be stronger as a 3-month pilot."

## Red flags

- No clear Zcash-specific benefit.
- Generic Web3/privacy hype.
- Budget mismatch.
- No forum post.
- Milestones based on activity rather than outputs.
- No evidence of ability to deliver.
- Large first request with weak track record.
- No maintenance plan.
- No KPIs.
- No user/community validation.
